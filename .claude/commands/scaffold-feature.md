# Scaffold a new backend feature: `$ARGUMENTS`

Generate the skeleton for a new **backend** feature module `$ARGUMENTS` under
`src/seedbank/` — one router, service, repository, schemas, and the three test
stubs — wired across all layers so it imports cleanly. This is the fast path; the
`add-endpoint` skill is the reference for doing it by hand or for understanding
each layer's contract. This command is purely backend: it does not touch
`frontend/` or `mobile/`.

## Before you start

1. **Validate the name: kebab-case plural noun** (e.g. `widgets`, `notifications`,
   `api-keys`). REST URLs are nouns and HTTP methods are the verbs, so reject
   verb-shaped names (`get-widgets`, `create-foo`). If it's singular, ask once
   whether they meant the plural.
2. Read `CLAUDE.md` and `.claude/skills/add-endpoint/SKILL.md` (especially the URL
   design section) — they hold the patterns the stubs must follow.
3. Check nothing under `src/seedbank/` already uses the name, to avoid a clash.
4. Decide if this is one feature or two. **Don't scaffold separate single/batch
   endpoints** — `POST /widgets` should accept one item *or* a list. Split only for
   a real reason (different auth, latency profile, or idempotency), because two
   near-identical routers drift apart and double the test surface.

## What you create

Derive the name forms:
- `feature_kebab` = the input (e.g. `widgets`)
- `feature_snake` = `feature_kebab.replace("-", "_")` (e.g. `api_keys`)
- `Feature` = singular PascalCase (e.g. `Widget`, `ApiKey`)

Files:

1. `src/seedbank/domain/<feature_snake_singular>.py` — framework-free dataclass entity stub.
2. `src/seedbank/schemas/<feature_snake>.py` — `<Feature>Create` and `<Feature>Read`
   Pydantic v2 schemas. **Don't** hand-roll a list/page schema — responses use the
   generic `Envelope[T]` / `Page[T]` from `seedbank.schemas.common`, so every client
   sees one stable shape.
3. `src/seedbank/infrastructure/db/repositories/<feature_snake>.py` —
   `<Feature>Repository` with `add`, `get_for_owner`, `list_for_owner`,
   `count_for_owner` skeletons. Repos return domain entities, not ORM rows.
4. `src/seedbank/services/<feature_snake>_service.py` — `<Feature>Service` with
   `create`, `get`, `list` skeletons. The service owns the transaction and raises
   `core.exceptions.*`; it must not import FastAPI.
5. `src/seedbank/api/v1/<feature_snake>.py` — `APIRouter` with POST + GET-by-id +
   GET list, RBAC via `current_user`. Single-resource routes use
   `response_model=Envelope[<Feature>Read]` and return `Envelope(data=...)`. The list
   route uses `response_model=Page[<Feature>Read]`, `?page=&page_size=` (1-indexed,
   max 200), and `paginate(items, total=..., page=..., page_size=...)`. Routers parse,
   call the service, and return — never import SQLAlchemy, never raise `HTTPException`.
6. `tests/unit/services/test_<feature_snake>_service.py` — happy path + quota/conflict
   + not-found. Mock the repo; never mock `AsyncSession`.
7. `tests/integration/db/test_<feature_snake>_repository.py` — ownership filter +
   soft-delete filter, against real Postgres (testcontainers).
8. `tests/e2e/test_<feature_snake>.py` — 201, 401, 404, 422 over the full HTTP path.
   Success bodies traverse `r.json()["data"]`; error bodies are RFC 9457 Problem
   Details — assert the Content-Type starts with `application/problem+json` and check
   `body["status"]` plus the machine-readable `body["code"]` (e.g. `forbidden`,
   `not_found`, `validation_error`).

Also wire it in:
- `src/seedbank/api/v1/__init__.py` — `from . import <feature_snake>` and
  `api_v1.include_router(<feature_snake>.router)`.
- `src/seedbank/api/deps.py` — DI factories `get_<feature_snake>_repo` and
  `get_<feature_snake>_service`.

**Don't add the SQLAlchemy ORM model yet.** Stop after the files above and tell the user:

> Append the `<Feature>ORM` class to `infrastructure/db/models.py`, then run
> `/new-migration "add <feature_kebab> table"` and review the generated revision.

Leaving the ORM + migration as a deliberate manual step keeps schema changes
reviewed rather than silently autogenerated alongside boilerplate.

## Stub content

Use the `add-endpoint` skill templates verbatim with the names substituted. Leave
service methods raising `NotImplementedError("scaffolded — implement me")` so they
fail loudly until written — but make sure every stub **imports cleanly** so
`make check` doesn't blow up on a missing symbol.

## After writing the files

```bash
ruff format src tests 2>/dev/null || true
ruff check src tests 2>/dev/null || true
python -c "import seedbank.api.v1.<feature_snake>" 2>&1 | head -50 || true
```

Report which files you created, what's still stubbed (`NotImplementedError`), and
the next concrete steps (implement the service, add the ORM model, run `/new-migration`).
